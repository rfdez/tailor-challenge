# 🎯 Client-Generated IDs

## 💡 Convention

Resource identifiers (UUIDs) are generated by the client — web frontend, mobile app, CLI script, or test runner — and sent to the backend as part of the request. The backend never generates IDs for new resources.

- The HTTP method for creating a resource is `PUT`, with the ID as part of the URL (`PUT /reservations/:id`).
- The ID must be a valid UUID. The API validates this at the route level (e.g. `z.uuid()`).
- The domain entity is constructed with a non-nullable `id` from the start — it never has a nullable or auto-generated identifier.
- Generation happens in the outermost layer: the client when possible, or a controller/adapter when the client cannot be modified (e.g. third-party integrations). It never happens in the database.

## 🏆 Benefits

- **Consistent domain model:** entities are always born with a valid identifier, eliminating nullable `id` fields in constructors and `toPrimitives()` typings.
- **Simplifies testing:** tests know the ID upfront without parsing dynamic responses, enabling precise assertions and easy comparison of full object graphs.
- **Idempotency:** `PUT` with a client-chosen ID is naturally idempotent — retrying the same request produces the same result without creating duplicates.
- **Security:** UUIDs do not reveal business volume (unlike sequential database IDs) and resist brute-force enumeration attacks.
- **Offline-first and optimistic UI:** the client can create resources with a definitive ID before the server confirms, enabling seamless offline-to-online synchronization.
- **Infrastructure decoupling:** the database has no logic for generating identifiers — it simply stores and indexes what it receives.

## 👀 Examples

### ✅ Good: Client sends a UUID, API validates and uses it directly

```typescript
// Route (src/app/api/reservations.ts)
const createReservationParamSchema = z.object({
  id: z.uuid(),
});

app.put(
  "/:id",
  sValidator("param", createReservationParamSchema),
  sValidator("json", createReservationBodySchema),
  async (c) => {
    const { id } = c.req.valid("param");
    // ...
    await creator.create(id, restaurantId, userId, date, time, partySize);
    return c.json(null, 201, { Location: `/reservations/${id}` });
  },
);
```

```typescript
// Domain (src/modules/reservations/domain/Reservation.ts)
export class Reservation {
  constructor(
    private readonly id: string, // never nullable
    // ...
  ) {}
}
```

### ❌ Bad: Database generates the ID

```typescript
// Route
app.post("/reservations", ... , async (c) => {
  const { restaurantId, date, time, partySize } = c.req.valid("json");

  const id = await reservationRepository.nextId();   // generated in infrastructure
  const reservation = Reservation.create(id, ...);
  await repository.save(reservation);

  return c.json({ id, ... });   // client must parse the response to get the ID
});
```

### ❌ Bad: Nullable ID in the domain

```typescript
export class Reservation {
  constructor(
    private readonly id: string | null, // nullable until database assigns it
    // ...
  ) {}
}
```
